Architecture

How the Ralph panel integrates with App state and broadcast events

Data Flow

RalphProcessManager
  ↓ broadcasts (RunId, WorkerEvent)
lib.rs::run() converts to RalphUpdate
  ↓ sends to ralph_tx
event_loop::run_event_loop()
  ↓ receives from app.ralph_rx
app.handle_ralph_update(update)
  ↓ delegates to
views.ralph_panel.on_update(update)
  ↓ mutates ActiveEntry / CompletedEntry
render() reads state on each tick

State Structure

pub struct RalphPanelView {
    active: Vec<ActiveEntry>,
    completed: Vec<CompletedEntry>,
    selected: usize,
}

struct ActiveEntry {
    run_id: String,
    loop_type: String,
    phase: String,
    progress: f64,       // 0.0-1.0
    message: String,
    started_at: Instant,
    log_lines: Vec<String>,
}

struct CompletedEntry {
    loop_type: String,
    success: bool,
    completed_at: Instant,
    summary: String,
}

Design Decisions

  • on_update() is a standalone method — NOT part of the View trait. Called directly from app.handle_ralph_update() since the view holds its own state
  • State lives in the view struct (not in App) — clean separation; App just routes events
  • Completed entry cleanup happens in render() via &mut self — no separate timer task needed
  • progress stored as 0.0–1.0 ratio (divided from percent); passed to ProgressBar::new(ratio, label) directly

Loop Entry State Machine

How RalphUpdate events transition loop entries between states

// RalphUpdate variants dispatched from app.handle_ralph_update()
pub enum RalphUpdate {
    Progress { id: String, phase: String, percent: f64 },
    LogLine  { id: String, line: String },
    Completed{ id: String },
    Failed   { id: String, reason: String },
}

// on_update state transitions:
// Progress (unknown id) → create new ActiveEntry (loop_type="unknown" until wired)
// Progress (known id)   → update phase + progress + message in-place
// LogLine  (known id)   → append to log_lines (capped at 50), update message
// Completed(known id)   → remove from active, push to completed (success=true)
// Failed   (known id)   → remove from active, push to completed (success=false)
// Completed/Failed older than 5s → removed by cleanup_expired() on each render

on_update implementation

pub fn on_update(&mut self, update: RalphUpdate) {
    match update {
        RalphUpdate::Progress { id, phase, percent } => {
            if let Some(entry) = self.active.iter_mut().find(|e| e.run_id == id) {
                entry.phase = phase.clone();
                entry.progress = percent / 100.0;
                entry.message = phase;
            } else {
                self.active.push(ActiveEntry {
                    run_id: id, loop_type: String::from("unknown"),
                    phase: phase.clone(), progress: percent / 100.0,
                    message: phase, started_at: Instant::now(),
                    log_lines: Vec::new(),
                });
            }
            self.clamp_selected();
        }
        RalphUpdate::Completed { id } => {
            if let Some(pos) = self.active.iter().position(|e| e.run_id == id) {
                let entry = self.active.remove(pos);
                let summary = entry.log_lines.last().cloned().unwrap_or_default();
                self.completed.push(CompletedEntry {
                    loop_type: entry.loop_type, success: true,
                    completed_at: Instant::now(), summary,
                });
                self.clamp_selected();
            }
        }
        // LogLine and Failed follow same pattern...
    }
}

5-Second Cleanup

fn cleanup_expired(&mut self) {
    let threshold = Duration::from_secs(COMPLETED_DISPLAY_SECS); // 5
    self.completed
        .retain(|e| e.completed_at.elapsed() < threshold);
}

// Called at start of every render() — no timer task, no tokio::spawn needed
// Works because render() takes &mut self and is called on every 250ms tick

handle_key in RalphPanelView

fn handle_key(&mut self, code: KeyCode, _modifiers: KeyModifiers) -> Option<Action> {
    match code {
        KeyCode::Char('j') | KeyCode::Down => {
            if !self.active.is_empty() && self.selected + 1 < self.active.len() {
                self.selected += 1;
            }
            None
        }
        KeyCode::Char('k') | KeyCode::Up => {
            if self.selected > 0 { self.selected -= 1; }
            None
        }
        KeyCode::Char('c') =>
            self.selected_run_id().map(Action::CancelRalphLoop),
        KeyCode::Enter =>
            self.selected_run_id().map(Action::RalphDetail),
        KeyCode::Esc =>
            Some(Action::NavigateBack),
        _ => None,
    }
}

Note: c is NOT in the global KeyMap, so it falls through to active_view_mut().handle_key() — the view receives it directly. j/k/Enter are in the KeyMap and are routed via the action delegation fix (next slide).

Action Routing Fix

ScrollDown/ScrollUp/Select now properly delegate to the active view for context-specific behavior

Problem

The global KeyMap intercepts j→ScrollDown before it reaches the view. Previously, handle_action(ScrollDown) was a no-op — so j never updated the Ralph panel's selected index.

Fix in app.rs

pub fn handle_action(&mut self, action: Action) {
    match action {
        // ...existing handlers...

        Action::ScrollDown => {
            if let Some(action) = self
                .active_view_mut()
                .handle_key(KeyCode::Down, KeyModifiers::NONE)
            {
                self.handle_action(action); // bubble up returned action
            }
        }
        Action::ScrollUp => {
            if let Some(action) = self
                .active_view_mut()
                .handle_key(KeyCode::Up, KeyModifiers::NONE)
            {
                self.handle_action(action);
            }
        }
        Action::Select => {
            if let Some(action) = self
                .active_view_mut()
                .handle_key(KeyCode::Enter, KeyModifiers::NONE)
            {
                self.handle_action(action);
            }
        }

        Action::CancelRalphLoop(_) | Action::RalphDetail(_) => {}
        // no-op until task 36 wires RalphProcessManager into App
    }
}

Full dispatch chain for j key

User presses 'j'
  → event_loop: map_key_to_action()
  → keybindings: KeyMap::resolve(Ralph, KeyCombo::plain('j'))
  → returns Action::ScrollDown
  → app.handle_action(ScrollDown)
  → active_view_mut().handle_key(KeyCode::Down, NONE)
  → RalphPanelView::handle_key(Down, NONE)
  → self.selected += 1
  → returns None
  → no further action, re-render on next tick

Backward compatible: all other stub views return None from handle_key, so the change is transparent for now.
