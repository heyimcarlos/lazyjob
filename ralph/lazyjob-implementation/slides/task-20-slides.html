Implementation

pub enum LoopType {
    JobDiscovery, CompanyResearch, ResumeTailor,
    CoverLetter, InterviewPrep,
}

impl LoopType {
    pub fn priority(&self) -> u8 {
        match self {
            CoverLetter     => 90,  // user-initiated, urgent
            ResumeTailor    => 85,
            InterviewPrep   => 70,
            CompanyResearch => 50,  // background enrichment
            JobDiscovery    => 30,  // periodic sweep
        }
    }

    pub fn concurrency_limit(&self) -> usize {
        match self {
            JobDiscovery => 1,  // serial to protect APIs
            CompanyResearch | InterviewPrep => 2,
            ResumeTailor | CoverLetter => 3,
        }
    }

    pub fn is_interactive(&self) -> bool {
        matches!(self, LoopType::InterviewPrep)
    }
}

Design Rationale

User-initiated = highest priority

When a user triggers a CoverLetter or ResumeTailor from the TUI, they're waiting for output. These must preempt background sweeps.

JobDiscovery concurrency = 1

Greenhouse and Lever have rate limits. Running parallel discovery sweeps risks 429 errors. Serial execution is safer.

Implementation

pub struct QueuedLoop {
    pub loop_type: LoopType,
    pub params: serde_json::Value,
    enqueued_at: Instant,  // private — tie-break: earlier = higher
}

impl Ord for QueuedLoop {
    fn cmp(&self, other: &Self) -> Ordering {
        let priority_ord = self.loop_type.priority().cmp(&other.loop_type.priority());
        if priority_ord != Ordering::Equal {
            return priority_ord;
        }
        // Tie-break: earlier enqueued has higher priority
        other.enqueued_at.cmp(&self.enqueued_at)
    }
}

pub struct LoopDispatch {
    heap: BinaryHeap<QueuedLoop>,
    cap: usize,  // always 20
}

impl LoopDispatch {
    pub fn enqueue(&mut self, lt: LoopType, params: Value) -> Result<()> {
        if self.heap.len() >= self.cap {
            return Err(RalphError::QueueFull(self.cap));
        }
        self.heap.push(QueuedLoop::new(lt, params));
        Ok(())
    }

    pub fn drain_next(&mut self) -> Option<QueuedLoop> { self.heap.pop() }
}

Key Properties

Hard cap = 20

Returns Err(QueueFull(20)) on the 21st enqueue. Never silently discards.

FIFO tie-break

Same-priority loops drain in insertion order (earlier-enqueued first).

O(log n) ops

BinaryHeap provides O(log n) push and pop. Fine for cap=20.

LoopScheduler

Cron expression parsing · single-fire guarantee · time-travel testable

How It Works

Parse expr → cron::Schedule
→
last_checked = now - 1ms
should_run(now)
→
Find next_tick = schedule.after(&last_checked).next()
→
if next_tick ≤ now → return true, advance last_checked

Implementation

pub struct LoopScheduler {
    schedule: cron::Schedule,
    last_checked: DateTime<Utc>,
}

impl LoopScheduler {
    pub fn new(expr: &str) -> Result<Self> {
        let schedule = cron::Schedule::from_str(expr)
            .map_err(|e| RalphError::CronParse(e.to_string()))?;
        let last_checked = Utc::now() - Duration::milliseconds(1);
        Ok(Self { schedule, last_checked })
    }

    pub fn should_run(&mut self, now: DateTime<Utc>) -> bool {
        if let Some(next_tick) = self.schedule.after(&self.last_checked).next()
            && next_tick <= now
        {
            self.last_checked = now;
            return true;
        }
        false
    }
}

Error Additions

Two new RalphError variants for Task 20

Updated RalphError

pub enum RalphError {
    // Existing:
    Decode(String),
    Io(#[from] std::io::Error),
    NotFound(String),

    // New in Task 20:
    CronParse(String),
    // Emitted by: LoopScheduler::new() when cron expression is invalid
    // Message: "cron parse error: {reason}"

    QueueFull(usize),
    // Emitted by: LoopDispatch::enqueue() when heap.len() >= cap
    // Message: "dispatch queue is full (capacity {cap})"
}

Error Behavior

CronParse

let r = LoopScheduler::new("not valid");
// Err(CronParse("failed to parse ..."))

QueueFull

// After 20 enqueues:
let r = dispatch.enqueue(lt, json!({}));
// Err(QueueFull(20))

Design Note

No #[from] on CronParse — cron errors are only produced in LoopScheduler::new(), so wrapping them as a String keeps the dependency surface minimal. The cron crate's error type doesn't need to leak into RalphError's public API.
