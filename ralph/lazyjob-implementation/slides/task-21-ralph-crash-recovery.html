Recovery Logic

How stale ralph runs are detected and recovered

The Problem

When the TUI crashes mid-run (OOM, kill signal, segfault), ralph subprocesses may still be marked “running” in the database. On next startup, these stale rows need cleanup.

Recovery Query

-- Find runs that were "running" but haven't updated in 30+ seconds
-- (if TUI just started, any "running" row > 30s old is from a past crash)
UPDATE ralph_loop_runs
   SET status = 'failed'
 WHERE status = 'running'
   AND started_at < now() - interval '30 seconds'

Rust Implementation

pub async fn recover_pending(&self) -> Result<usize> {
    let result = sqlx::query(
        "UPDATE ralph_loop_runs SET status = 'failed'
         WHERE status = 'running'
           AND started_at < now() - interval '30 seconds'",
    )
    .execute(&self.pool)
    .await?;
    Ok(result.rows_affected() as usize)
}

The 30-second threshold is conservative — any legitimate “running” row from this TUI session will be less than 1 second old at startup time.

TUI Wiring

How crash recovery integrates into App startup

Updated run() in lazyjob-tui/src/lib.rs

pub async fn run(config: Arc<Config>) -> anyhow::Result<()> {
    let (_ralph_tx, ralph_rx) = broadcast::channel::<RalphUpdate>(64);

    let app = match Database::connect(&config.database_url).await {
        Ok(db) => {
            let repo = RalphLoopRunRepository::new(db.pool().clone());
            let recovered = repo.recover_pending().await.unwrap_or(0);
            if recovered > 0 {
                tracing::info!("Recovered {recovered} stale ralph loop runs");
            }
            App::new(config, ralph_rx).with_pool(db.pool().clone())
        }
        Err(err) => {
            // Graceful fallback: TUI still launches without a DB
            tracing::warn!("Could not connect to database at startup: {err}");
            App::new(config, ralph_rx)
        }
    };

    event_loop::run_event_loop(app).await
}

Key Decisions

Design choices made during implementation

No migration 002 needed

The ralph_loop_runs table was already created in 001_initial_schema.sql by a prior iteration. Creating a 002 migration would fail on first run (table already exists). The existing schema matched all requirements perfectly.

recovery in run() not App::new()

App::new() is synchronous (for test convenience). Database connection and crash recovery happen in the async run() function before creating App. This pattern avoids making App::new() async and breaking all test code.

Graceful fallback when DB unavailable

If Database::connect() fails (e.g., Postgres not running), the TUI still launches with pool: None. A warning is logged. This lets the TUI work in offline/demo mode without crashing at startup.

30-second stale threshold

Any “running” row with started_at < now() - 30s at TUI startup must be from a previous crashed session — this TUI session is only milliseconds old at recovery time. 30s is conservative enough to avoid false positives.

RalphLoopRunStatus as snake_case TEXT

Follows the same pattern as ApplicationStage — stored as TEXT in PG, converted via as_str() and FromStr. This avoids PG custom enum type registration complexity and keeps migrations simple.
