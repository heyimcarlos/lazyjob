Architecture Decision

Avoiding the circular dependency trap — same solution as Task 25's Embedder trait

The Circular Dependency Problem

❌ Cannot do: lazyjob-core importing lazyjob-llm::LlmProvider


Because lazyjob-llm already depends on lazyjob-core (for sqlx token usage logging). A cycle would break the build.

Solution: Local Trait Pattern

Following the exact same approach established in Task 25's Embedder trait:

lazyjob-core
  • Defines local Completer trait
  • CompanyResearcher holds Arc<dyn Completer>
  • No lazyjob-llm dependency
  • Fully testable with MockCompleter
lazyjob-cli (bridge)
  • Defines LlmProviderCompleter
  • Wraps Box<dyn LlmProvider>
  • Implements Completer trait
  • Passed as Arc<dyn Completer>
lazyjob-llm
  • LlmProvider trait unchanged
  • No knowledge of Completer
  • AnthropicProvider / OllamaProvider
  • Already depends on lazyjob-core

Dependency Graph

lazyjob-llm
→
lazyjob-core
←
lazyjob-cli
→
lazyjob-llm

The CLI layer owns the bridge — it can import both crates freely.

The Completer Trait

A single-method async trait — minimal surface, maximum testability

#[async_trait]
pub trait Completer: Send + Sync {
    async fn complete(&self, system: &str, user: &str) -> Result<String>;
}

Test Implementation (MockCompleter)

struct MockCompleter {
    response: String,
}

#[async_trait]
impl Completer for MockCompleter {
    async fn complete(&self, _system: &str, _user: &str) -> Result<String> {
        Ok(self.response.clone())
    }
}

Production Implementation (LlmProviderCompleter)

struct LlmProviderCompleter {
    provider: Box<dyn LlmProvider>,
}

#[async_trait]
impl Completer for LlmProviderCompleter {
    async fn complete(&self, system: &str, user: &str) -> Result<String> {
        let messages = vec![
            ChatMessage::System(system.into()),
            ChatMessage::User(user.into()),
        ];
        self.provider
            .complete(messages, CompletionOptions::default()).await
            .map(|r| r.content)
            .map_err(|e| CoreError::Parse(e.to_string()))
    }
}

Design Notes

Arc<dyn Completer>

Same pattern as MatchScorer's Arc<dyn Embedder> from Task 25. Enables shared ownership across async tasks and easy test injection.

reqwest::Client injected

HTTP client passed in rather than constructed internally — allows tests to share a client, and production code to configure timeouts/TLS once at startup.

Usage from CLI

let provider = LlmBuilder::from_config(&config, &creds)?;
let completer = Arc::new(LlmProviderCompleter { provider });
let client    = reqwest::Client::new();
let researcher = CompanyResearcher::new(completer, client);
let data = researcher.enrich(&company_id, db.pool()).await?;

enrich() — Step by Step

6-step pipeline from company_id to enriched DB record

1. CompanyRepository::find_by_id() → CoreError::NotFound if missing
↓
2. company.website.as_ref() → CoreError::Validation if no URL
↓
3. reqwest.get(website).send().await → HTML body
↓
4. strip_html(html) → plain text, truncate to 3000 chars
↓
5. completer.complete(system, user) → LLM JSON response
↓
6. extract_json_from_response() → update Company + repo.update()

LLM Prompt Design

let system = "You are a company research assistant. Extract factual company information \
              from website content and return it as a single JSON object.";

let user = format!("Extract company information ... return ONLY a JSON object with:
{{'industry': ..., 'size': ..., 'tech_stack': [...],
  'culture_keywords': [...], 'recent_news': [...] }}

Website content:\n{content}");

Robust JSON Extraction

fn extract_json_from_response(response: &str) -> Result<EnrichmentData> {
    let json_start = trimmed.find('{')?;  // handles LLM preamble text
    let json_end   = trimmed.rfind('}')?; // finds outermost object
    serde_json::from_str(&trimmed[json_start..=json_end])
}

This handles the common case where an LLM adds preamble like "Here is the JSON:" before the actual JSON object.

Enrichment Status Badge

Visual indicator in JobsListView for company research status

Badge Function

pub fn enrichment_badge(industry: Option<&str>) -> &'static str {
    if industry.is_some() { "[E]" } else { "[ ]" }
}

Badge Appearance

[E] — Company has been researched (industry is populated)


[ ] — Company not yet enriched (industry is None)
