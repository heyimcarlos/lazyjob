Architecture Decision

🔄 Circular Dependency Problem

// Adding the reverse would create a cycle:
// lazyjob-core → lazyjob-llm → lazyjob-core ❌

✅ Solution: Local Embedder Trait

🦀 Implementation

#[derive(Default)]
pub struct GhostDetector {
    pub duplicate_description: bool,
    pub high_application_count: bool,
}

impl GhostDetector {
    pub fn score(&self, job: &Job) -> GhostScore {
        let mut score: u8 = 0;
        let age_days = (Utc::now() - job.discovered_at).num_days();
        if age_days > 60 { score = score.saturating_add(3); }
        if GENERIC_TITLES.iter().any(|&t| t == job.title.to_lowercase()) {
            score = score.saturating_add(2);
        }
        // ... 5 more signals
        GhostScore { score }
    }
}

impl GhostScore {
    pub fn is_likely_ghost(&self) -> bool { self.score >= 5 }
}

MatchScorer

Cosine similarity between life sheet profile and job descriptions

📐 Cosine Similarity Formula

pub fn cosine_similarity(a: &[f32], b: &[f32]) -> f32 {
    if a.len() != b.len() || a.is_empty() { return 0.0; }

    let dot: f32  = a.iter().zip(b).map(|(x, y)| x * y).sum();
    let norm_a: f32 = a.iter().map(|x| x * x).sum::<f32>().sqrt();
    let norm_b: f32 = b.iter().map(|x| x * x).sum::<f32>().sqrt();

    if norm_a == 0.0 || norm_b == 0.0 { return 0.0; }
    (dot / (norm_a * norm_b)).clamp(-1.0, 1.0)
}

Migration 003

job_embeddings table — BYTEA storage for f32 vectors

🗄️ SQL Schema

-- crates/lazyjob-core/migrations/003_job_embeddings.sql
CREATE TABLE job_embeddings (
    job_id      UUID PRIMARY KEY
                     REFERENCES jobs(id) ON DELETE CASCADE,
    embedding   BYTEA NOT NULL,
    embedded_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

💾 BYTEA Serialization

// Vec<f32> → raw little-endian bytes (4 bytes per f32)
fn embedding_to_bytes(embedding: &[f32]) -> Vec<u8> {
    embedding.iter().flat_map(|f| f.to_le_bytes()).collect()
}

// Reverse: split bytes into 4-byte chunks, parse each f32
fn bytes_to_embedding(bytes: &[u8]) -> Vec<f32> {
    bytes.chunks_exact(4)
        .map(|c| f32::from_le_bytes([c[0], c[1], c[2], c[3]]))
        .collect()
}
Why BYTEA over REAL[]? BYTEA is a contiguous blob — no array overhead or element type casting. For embedding dimensions of 768-1536 elements, BYTEA is ~4x more compact and faster to serialize/deserialize than PG array literals. No pgvector extension needed for the current scale (~100s-1000s of jobs).

🔄 Upsert Pattern

INSERT INTO job_embeddings (job_id, embedding)
VALUES ($1, $2)
ON CONFLICT (job_id) DO UPDATE
SET embedding = EXCLUDED.embedding,
    embedded_at = now()

Idempotent: re-embedding a job updates the stored vector atomically.
