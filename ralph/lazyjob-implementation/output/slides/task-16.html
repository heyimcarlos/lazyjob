OpenAiProvider Implementation

Uses the official async-openai 0.34 SDK with builder-pattern request construction.

Construction

pub struct OpenAiProvider {
    client: Client<OpenAIConfig>,
    default_model: String,
    embedding_model: String,
}

impl OpenAiProvider {
    // Direct API key
    pub fn new(api_key: impl Into<String>) -> Self {
        let config = OpenAIConfig::new().with_api_key(api_key.into());
        Self { client: Client::with_config(config), .. }
    }

    // From keyring CredentialManager
    pub fn from_credentials(creds: &CredentialManager) -> Result<Self> {
        let secret = creds.get_api_key("openai")?
            .ok_or_else(|| LlmError::Auth("no key for 'openai'".into()))?;
        Ok(Self::new(secret.expose_secret().to_owned()))
    }
}

OllamaProvider Implementation

Uses ollama-rs 0.3 to communicate with a local Ollama server (default localhost:11434).

Construction

pub struct OllamaProvider {
    client: Ollama,
    default_model: String,   // "llama3.2:latest"
    embedding_model: String, // "nomic-embed-text"
}

impl OllamaProvider {
    // localhost:11434
    pub fn new() -> Self { .. Ollama::new(DEFAULT_HOST, DEFAULT_PORT) .. }

    // Custom endpoint
    pub fn with_host_port(host: impl Into<String>, port: u16) -> Self {
        Self { client: Ollama::new(host.into(), port), .. }
    }
}

// ollama-rs has its own ChatMessage — aliased to avoid conflict

    match msg {
        ChatMessage::System(c)    => OllamaMessage::system(c),
        ChatMessage::User(c)      => OllamaMessage::user(c),
        ChatMessage::Assistant(c) => OllamaMessage::assistant(c),
    }
}

Key Challenges Encountered

1 · async-openai 0.34 Feature Gating

Problem: Client, config::OpenAIConfig and builder types not visible by default. They're behind the _api feature, which must be activated indirectly. 
# Wrong — only types, no HTTP client
async-openai = "0.34"

# Correct — chat-completion and embedding activate _api feature
async-openai = { version = "0.34", features = ["chat-completion", "embedding"] }

2 · Type Paths Changed in 0.34

Problem: Builder types are in submodules, not re-exported at crate root. 
// Wrong in 0.34:
use async_openai::types::CreateChatCompletionRequestArgs;

// Correct:
use async_openai::types::chat::CreateChatCompletionRequestArgs;
use async_openai::types::embeddings::CreateEmbeddingRequestArgs;

3 · ollama-rs native-tls dependency

Problem: Default features pull in native-tls → openssl-sys, which requires libssl-dev headers (not installed on this system). 
# Wrong — includes native-tls/openssl-sys
ollama-rs = "0.3"

# Correct — use rustls, disable default native-tls
ollama-rs = { version = "0.3", default-features = false,
              features = ["stream", "rustls"] }

4 · Ambiguous provider_name() method

Problem: Both LlmProvider and EmbeddingProvider define provider_name(). In tests, provider.provider_name() is ambiguous. 
// Wrong — ambiguous in test code:
assert_eq!(provider.provider_name(), "openai");

// Correct — use qualified syntax:
use crate::provider::LlmProvider;
assert_eq!(LlmProvider::provider_name(&provider), "openai");

Key Design Decisions

Both providers implement both traits

Decision: Both OpenAiProvider and OllamaProvider implement both LlmProvider and EmbeddingProvider.

Rationale: OpenAI has both chat and embeddings APIs. Ollama supports both locally. This makes them self-contained — callers can use either provider for all LLM needs without importing a separate embedding provider.

No retry backoff for OpenAI/Ollama

Decision: No exponential backoff implemented in this iteration (unlike AnthropicProvider).

Rationale: The async-openai SDK handles retries internally. Ollama is local (no rate limiting needed). Task 17 (ProviderRegistry) will add unified retry policy at the registry layer.

rustls over native-tls for Ollama

Decision: ollama-rs configured with default-features = false, features = ["stream", "rustls"].

Rationale: System doesn't have libssl-dev. The existing reqwest workspace dep already uses rustls-tls. Using rustls everywhere is consistent and avoids mixed TLS stacks.

Type alias for name conflict

Decision: use ollama_rs::generation::chat::ChatMessage as OllamaMessage.

Rationale: Our own crate::message::ChatMessage and ollama-rs's ChatMessage would collide in the same file. Aliasing is idiomatic Rust.
