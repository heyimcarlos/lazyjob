Types

ChatMessage — Enum Approach

Using an enum (not a struct with role: String) gives type-safe pattern matching and prevents constructing invalid messages.

#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "role", content = "content", rename_all = "lowercase")]
pub enum ChatMessage {
    System(String),
    User(String),
    Assistant(String),
}

impl ChatMessage {
    pub fn role(&self) -> &str { /* "system" | "user" | "assistant" */ }
    pub fn content(&self) -> &str { /* inner string */ }
}

TokenUsage — Auto-computes Total

pub struct TokenUsage {
    pub prompt_tokens: u32,
    pub completion_tokens: u32,
    pub total_tokens: u32,
}

impl TokenUsage {
    pub fn new(prompt: u32, completion: u32) -> Self {
        Self { prompt_tokens: prompt, completion_tokens: completion,
               total_tokens: prompt + completion }
    }
}

use async_trait::async_trait;

use crate::error::Result;
use crate::message::{ChatMessage, CompletionOptions, LlmResponse};

#[async_trait]
pub trait LlmProvider: Send + Sync {
    fn provider_name(&self) -> &str;
    fn model_name(&self) -> &str;

    async fn complete(
        &self,
        messages: Vec<ChatMessage>,
        opts: CompletionOptions,
    ) -> Result<LlmResponse>;
}

#[async_trait]
pub trait EmbeddingProvider: Send + Sync {
    fn provider_name(&self) -> &str;

    async fn embed(&self, text: &str) -> Result<Vec<f32>>;
}
Why async_trait? Rust 2024 stabilized async fn in traits for static dispatch, but dyn Trait still requires boxed futures. The #[async_trait] macro rewrites methods to return Pin<Box<dyn Future + Send>>, enabling dynamic dispatch. This is the proven solution with no runtime overhead beyond the existing Box.

Key Decisions

1. async_trait vs explicit Pin<Box<dyn Future>>

Rust 2024 stabilized async fn in traits for static dispatch, but dyn Trait still needs boxed futures. The async_trait macro is the proven, zero-friction solution — it rewrites methods to return Pin<Box<dyn Future + Send>>. This is what the spec recommends and what the ecosystem uses.

2. ChatMessage as Enum (not Struct)

A struct ChatMessage { role: String, content: String } would allow invalid states (e.g., role = "foo"). The enum variant approach makes invalid messages a compile error and enables exhaustive pattern matching in provider implementations when building API payloads.

3. CompletionOptions uses Option<T> for all configurable fields

Callers override only what they need. The Default impl provides sensible values (temperature=0.7, max_tokens=4096). Providers check opts.model.unwrap_or(self.default_model) — a clean override pattern.

4. MockLlmProvider::with_content() convenience constructor

The common test pattern is "make the LLM return this specific text." The with_content("...") helper eliminates 5 lines of boilerplate per test: creating TokenUsage, LlmResponse, then MockLlmProvider::new(). All 14 future pipeline tests will use this.

5. Send + Sync bounds on both traits

LLM calls happen in async contexts that may be multi-threaded (tokio work-stealing). Without Send + Sync, Box<dyn LlmProvider> cannot be held across .await points or moved between tasks. These bounds are required for practical use.

Rejected Alternative: Trait-per-capability (Complete, Embed, Stream as separate traits)

Would require callers to deal with multiple trait objects. The two-trait design (LlmProvider for chat, EmbeddingProvider for vectors) matches the actual distinction: some providers don't support embeddings (Anthropic). This is the right split.
