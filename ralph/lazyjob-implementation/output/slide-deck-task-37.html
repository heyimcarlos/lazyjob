First Name: "First Name", "first_name", "FirstName"

Company:    "Company", "company", "Organization"
Position:   "Position", "position", "Title", "Job Title"
URL:        "URL", "Profile URL", "linkedin_url"

pub fn parse_linkedin_csv<R: Read>(reader: R) -> Result<Vec<Contact>>

Behavior

  • Requires at least First Name or Last Name header
  • Skips rows with empty names

  • Handles flexible column counts (extra/missing columns)
  • Sets source = "linkedin_csv" on all imported contacts

WarmPath Type

pub struct WarmPath {
    pub contact_name: String,
    pub contact_role: Option<String>,
    pub contact_email: Option<String>,
}

Matching Logic

  • Pure function - no DB queries, works on in-memory data
  • Case-insensitive company name matching
  • Returns empty if job has no company_name
  • Filters contacts whose current_company matches job's company_name

Example

// Contact: Alice Smith @ "ACME CORP"
// Job: Senior Engineer @ "acme corp"

update_warm_paths()

pub fn update_warm_paths(&mut self) {
    let job = match self.views.job_detail.job() {
        Some(j) => j.clone(),
        None => return,
    };
    let warm_paths = networking::warm_paths_for_job(
        self.views.contacts.contacts(), &job
    );
    self.views.job_detail.set_warm_paths(warm_paths);
}

Key Decisions

Extended Contact rather than new type

Added current_company and source fields to existing Contact domain type rather than creating parallel networking-specific types. Minimizes duplication while enabling CSV import and warm path matching.

Text-based company matching

Warm paths match by current_company text (case-insensitive) rather than company_id FK. CSV imports have company names, not UUIDs. Simple string comparison is sufficient for MVP.

Partial unique index on email

Created WHERE email IS NOT NULL unique index to support upsert deduplication while allowing multiple contacts without email addresses.

Pure function for warm path computation

warm_paths_for_job() operates on in-memory slices rather than querying the database. Contacts are already loaded in the TUI, so no additional DB round-trips needed.

500-contact limit on initial load

load_contacts() uses Pagination { limit: 500, offset: 0 }. Most users will have < 500 contacts imported. Future tasks can add pagination/search.
