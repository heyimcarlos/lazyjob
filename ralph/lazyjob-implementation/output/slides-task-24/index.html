DiscoveryService

Core implementation — parallel fan-out with progress events

pub struct DiscoveryStats {
    pub jobs_found: usize,
    pub jobs_new: usize,
    pub jobs_updated: usize,
    pub errors: usize,
}

impl std::ops::Add for DiscoveryStats { // enables fold aggregation
    type Output = Self;
    fn add(self, rhs: Self) -> Self { /* sum each field */ }
}

pub struct DiscoveryService;

impl DiscoveryService {
    pub async fn run_discovery(
        pool: &PgPool,
        sources: Vec<SourceConfig>,
        progress_tx: Option<mpsc::Sender<DiscoveryProgress>>,
    ) -> Result<DiscoveryStats> {
        let futs: Vec<_> = sources
            .into_iter()
            .map(|cfg| {
                let pool = pool.clone();
                let tx = progress_tx.clone();
                async move { discover_one(pool, cfg, tx).await }
            })
            .collect();

        let results = join_all(futs).await;

        let stats = results
            .into_iter()
            .fold(DiscoveryStats::default(), |acc, r| acc + r);

        Ok(stats)
    }
}

discover_one — per-source logic

async fn discover_one(pool: PgPool, cfg: SourceConfig, tx: Option<..>) -> DiscoveryStats {
    let client: Box<dyn JobSource> = match cfg.source.as_str() {
        "greenhouse" => Box::new(GreenhouseClient::new()),
        "lever"      => Box::new(LeverClient::new()),
        other => { /* error + return */ }
    };

    let jobs = client.fetch_jobs(&cfg.company_id).await?;
    let repo = JobRepository::new(pool);

    for job in &jobs {
        match repo.upsert_discovered(job).await {
            Ok(true)  => jobs_new += 1,
            Ok(false) => jobs_updated += 1,
            Err(e)   => { tracing::warn!(...); errors += 1; }
        }
    }
    DiscoveryStats { jobs_found: jobs.len(), jobs_new, jobs_updated, errors }
}

upsert_discovered Return Type

Using PostgreSQL's xmax system column to detect new vs updated rows

The problem

The old upsert_discovered(&self, job: &Job) -> Result<()> couldn't tell us whether the row was newly inserted or an update. The discovery service needs this to populate jobs_new and jobs_updated in DiscoveryStats.

Solution: PostgreSQL xmax trick

After an ON CONFLICT DO UPDATE, RETURNING (xmax = 0) AS is_new returns true for fresh inserts and false for updates. xmax stores the XID of the deleting transaction — zero means no prior version existed.

pub async fn upsert_discovered(&self, job: &Job) -> Result<bool> {
    let row = sqlx::query_as::<_, (bool,)>(
        "INSERT INTO jobs (...)
         VALUES (...)
         ON CONFLICT (source, source_id)
         WHERE source IS NOT NULL AND source_id IS NOT NULL
         DO UPDATE SET
             title       = EXCLUDED.title,
             company_name = EXCLUDED.company_name,
             location    = EXCLUDED.location,
             url         = EXCLUDED.url,
             description = EXCLUDED.description,
             updated_at  = now()
         RETURNING (xmax = 0) AS is_new",
    )
    .bind(...) // 15 fields
    .fetch_one(&self.pool).await?;

    Ok(row.0)  // true = newly inserted, false = updated
}

xmax semantics

  • INSERT → xmax = 0 — no prior transaction owned this tuple
  • UPDATE → xmax ≠ 0 — the old XID is stored to mark the deleted version
  • After DO UPDATE — xmax reflects whether the tuple existed before this statement

Verified by integration tests

  • upsert_discovered_returns_true_for_new — first insert returns true
  • upsert_discovered_returns_false_for_update — second upsert with same source+source_id returns false
  • Uses unique source_id per test run to avoid cross-test interference

Implementation — tokio::join! for concurrent progress printing

async fn handle_ralph(args: RalphArgs, db: &Database) -> Result<()> {
    let (RalphCommand::JobDiscovery { source, company_id }) = args.command;
    let (tx, mut rx) = tokio::sync::mpsc::channel(64);
    let sources = vec![SourceConfig { source, company_id }];

    let discover_fut = DiscoveryService::run_discovery(pool, sources, Some(tx));
    let print_fut = async {
        while let Some(p) = rx.recv().await {
            println!("[{}] {}: {}", p.source, p.company_id, p.message);
        }
    };

    let (stats, _) = tokio::join!(discover_fut, print_fut);
    println!("Discovery complete: {} found, {} new...", stats?.jobs_found, ...);
}

Learning Test

Proving futures::future::join_all works before depending on it

The challenge: async block opaque types

Each async { ... } block in Rust has a unique opaque type. You cannot put multiple inline async blocks into a Vec without boxing — even if they're identical, the type system considers them different. This is a subtlety worth proving before using join_all in production code.

// learning test: verifies library behavior
#[tokio::test]
async fn futures_join_all_collects_from_parallel_futures() {
    // Each call to the same async fn produces the same concrete type — safe for Vec
    async fn const_fut(n: usize) -> usize { n }

    let futs = vec![const_fut(1), const_fut(2), const_fut(3)];
    let results = join_all(futs).await;
    assert_eq!(results, vec![1, 2, 3]);
}

What it proves

  • join_all from futures 0.3 collects results in order
  • Named async functions produce a single concrete type — usable in homogeneous Vec
  • Inline async blocks are NOT interchangeable even if identical

// error[E0308]: mismatched types
// no two async blocks have the same type

    async { 2usize },  // type B ≠ A!
