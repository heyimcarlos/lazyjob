Rate Limiter

Interior mutability pattern — works with &self for trait objects

💡 Key insight: std::sync::Mutex (not async Mutex) is used because the lock is held for nanoseconds, never across an await point. Safe and fast.

Implementation

pub struct RateLimiter {
    interval:  Duration,
    last_call: Mutex<Option<Instant>>,   // interior mutability
}

impl RateLimiter {
    pub fn new(requests_per_second: u32) -> Self {
        let nanos = 1_000_000_000u64 / requests_per_second.max(1) as u64;
        Self {
            interval:  Duration::from_nanos(nanos),
            last_call: Mutex::new(None),
        }
    }

    pub async fn wait(&self) {
        let sleep_duration = {
            let mut guard = self.last_call.lock().expect("mutex poisoned");
            let now = Instant::now();
            let sleep = match *guard {
                Some(last) => {
                    let elapsed = now.duration_since(last);
                    if elapsed < self.interval { Some(self.interval - elapsed) }
                    else { None }
                }
                None => None,
            };
            *guard = Some(now);
            sleep                            // ← lock released here, before await
        };
        if let Some(dur) = sleep_duration {
            tokio::time::sleep(dur).await;  // ← no lock held during await
        }
    }
}

Mapping to Job Domain Type

fn into_job(self, board_token: &str) -> Job {
    let description = self.content.as_deref().map(strip_html);  // HTML → plain text
    let mut job = Job::new(self.title);
    job.source      = Some("greenhouse".to_string());
    job.source_id   = Some(self.id.to_string());     // i64 → String
    job.company_name = Some(board_token.to_string());  // board token = company
    job.location    = self.location.and_then(|l| l.name);
    job.url         = self.absolute_url;
    job.description = description;
    job
}

Mapping to Job Domain Type

fn into_job(self, company_id: &str) -> Job {
    let description = self.description.as_deref().map(strip_html);
    let mut job = Job::new(self.text);     // Lever uses "text" not "title"
    job.source      = Some("lever".to_string());
    job.source_id   = Some(self.id);        // already a String UUID
    job.company_name = Some(company_id.to_string());
    job.url         = self.hosted_url;
    job.description = description;
    if let Some(cats) = self.categories {
        job.location = cats.location;       // nested in categories object
    }
    job
}

HTML Stripping

Job APIs return HTML-formatted descriptions — we strip all tags for plain text storage

Implementation

use std::collections::HashSet;

pub(crate) fn strip_html(html: &str) -> String {
    ammonia::Builder::new()
        .tags(HashSet::new())    // allow NO HTML tags → all tags stripped
        .clean(html)
        .to_string()
}
💡 ammonia strips tags but preserves text content and decodes HTML entities. Using Builder with an empty tag set is the idiomatic way to convert HTML to plain text.

Learning Test — Proves Library Behavior

Migration & Upsert

Idempotent job ingestion — same job from same source never duplicates

Migration 002 — Partial Unique Index

-- crates/lazyjob-core/migrations/002_unique_job_source.sql
-- Partial unique index on (source, source_id) for ON CONFLICT upsert deduplication.
-- NULL values are excluded so manually-entered jobs bypass the constraint.
CREATE UNIQUE INDEX idx_jobs_source_id_unique
    ON jobs(source, source_id)
    WHERE source IS NOT NULL AND source_id IS NOT NULL;
💡 Why partial? Manually entered jobs have source = NULL. A full unique index would prevent two manual jobs from being added — a partial index on non-NULL values is the correct design.

upsert_discovered() in JobRepository

pub async fn upsert_discovered(&self, job: &Job) -> Result<()> {
    sqlx::query("INSERT INTO jobs (...) VALUES (...)
         ON CONFLICT (source, source_id)
         WHERE source IS NOT NULL AND source_id IS NOT NULL
         DO UPDATE SET
             title        = EXCLUDED.title,
             company_name = EXCLUDED.company_name,
             location     = EXCLUDED.location,
             url          = EXCLUDED.url,
             description  = EXCLUDED.description,
             updated_at   = now()")
    .bind(job.id)
    // ... bind all fields ...
    .execute(&self.pool)
    .await?;
    Ok(())
}

First discovery

INSERT succeeds, new row created with the job's new UUID.

Re-discovery (same source + source_id)

UPDATE fires: title, location, description, url refreshed. UUID unchanged.
